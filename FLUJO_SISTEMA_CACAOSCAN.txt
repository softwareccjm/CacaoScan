================================================================================
                          INTRODUCCIÓN
================================================================================

CacaoScan es un sistema que permite analizar granos de cacao usando inteligencia
artificial. El sistema toma fotos de granos de cacao y automáticamente calcula
sus dimensiones (alto, ancho, grosor) y peso sin necesidad de medirlos
físicamente.

El sistema está compuesto por dos partes principales que se comunican entre sí:
un frontend (interfaz de usuario) y un backend (procesamiento y lógica de
negocio). Adicionalmente, tiene un módulo de machine learning que es quien hace
las predicciones.

================================================================================
                    COMPONENTES PRINCIPALES
================================================================================

FRONTEND:
---------
La parte visible del sistema, lo que el usuario ve y usa. Está construida con
Vue.js, que es un framework moderno para crear interfaces web interactivas.
Permite que el usuario:
- Inicie sesión
- Suba imágenes de granos de cacao
- Vea los resultados del análisis
- Gestione fincas y lotes
- Genere reportes

BACKEND:
--------
El servidor que procesa todas las peticiones. Está construido con Django, que
es un framework de Python para crear aplicaciones web robustas. Este backend:
- Recibe las imágenes del frontend
- Las procesa con los modelos de IA
- Guarda los resultados en una base de datos
- Gestiona la autenticación de usuarios
- Coordina todas las operaciones del sistema

BASE DE DATOS:
--------------
Usa PostgreSQL, una base de datos muy confiable. Aquí se guardan:
- Usuarios del sistema
- Imágenes subidas
- Resultados de los análisis
- Información de fincas y lotes
- Historial de entrenamientos

MÁQUINA DE APRENDIZAJE:
-----------------------
El "cerebro" del sistema. Está dividido en dos partes:
1. Segmentación: Encuentra el grano de cacao en la imagen y lo aísla del fondo
2. Predicción: Calcula las dimensiones y peso basándose en el grano aislado

COLA DE TAREAS:
---------------
Para trabajos largos (como entrenar los modelos de IA), el sistema usa Celery,
que permite ejecutar tareas en segundo plano sin bloquear la interfaz. Redis
actúa como el mensajero que coordina estas tareas.

================================================================================
                    FLUJO GENERAL DEL SISTEMA
================================================================================

1. INICIO DE SESIÓN
--------------------

Cuando un usuario quiere usar el sistema, primero debe iniciar sesión.

El usuario ingresa su nombre de usuario y contraseña en la interfaz (frontend).
El frontend envía estas credenciales al backend mediante una petición HTTP.

El backend verifica las credenciales consultando la base de datos. Si son
correctas, genera un token de autenticación (JWT) que funciona como una
"llave" temporal que permite al usuario hacer peticiones al sistema.

Este token se guarda en el navegador del usuario y se envía automáticamente
en cada petición posterior para identificar al usuario.

Si las credenciales son incorrectas, el sistema muestra un error y no permite
el acceso.

Herramientas usadas:
- Frontend: Axios para enviar peticiones HTTP
- Backend: Django REST Framework con JWT (JSON Web Tokens)
- Base de datos: PostgreSQL para verificar credenciales

Limitaciones:
- Los tokens tienen tiempo de expiración, por lo que el usuario debe volver a
  iniciar sesión después de cierto tiempo
- Si el backend no está corriendo, no es posible iniciar sesión


2. SUBIDA Y ANÁLISIS DE IMÁGENES
----------------------------------

Este es el corazón del sistema. El usuario sube una o varias imágenes de granos
de cacao y el sistema las analiza automáticamente.

Paso a paso:

A) SUBIDA DE IMÁGENES
El usuario selecciona las imágenes desde su computadora. El frontend crea un
formulario especial que incluye las imágenes y metadatos (como nombre del lote,
finca, fecha, etc.).

B) ENVÍO AL BACKEND
El frontend envía todo esto al backend mediante una petición HTTP. Como las
imágenes pueden ser grandes y múltiples, el sistema tiene un tiempo de espera
de hasta 2 minutos para procesar todo un lote de imágenes.

C) PROCESAMIENTO CON YOLO (SEGMENTACIÓN)
El backend recibe las imágenes y comienza a procesarlas una por una. El primer
paso es usar un modelo de inteligencia artificial llamado YOLO, que está
especialmente entrenado para encontrar granos de cacao en imágenes.

YOLO analiza la imagen y genera una "máscara" que identifica exactamente dónde
está el grano de cacao. Si YOLO no encuentra el grano con la confianza inicial,
el sistema intenta con umbrales más bajos para asegurarse de encontrarlo.

Una vez identificado el grano, se crea una nueva imagen donde el grano se
mantiene pero el fondo se vuelve transparente. Esto ayuda a que los modelos de
predicción se concentren solo en el grano y no se distraigan con el fondo.

D) PREDICCIÓN CON MODELOS DE REGRESIÓN
Con el grano aislado, ahora se usan modelos de deep learning (específicamente
ResNet18, que es una arquitectura de red neuronal muy efectiva) para predecir
las dimensiones.

El sistema tiene 4 modelos separados, uno para cada medida:
- Alto (en milímetros)
- Ancho (en milímetros)
- Grosor (en milímetros)
- Peso (en gramos)

Cada modelo fue entrenado con cientos de imágenes donde se conocían las medidas
reales, por lo que "aprendió" a asociar características visuales del grano con
sus dimensiones.

El sistema también calcula un nivel de confianza para cada predicción. Esto se
hace ejecutando el modelo varias veces con pequeñas variaciones y analizando qué
tan consistentes son los resultados. Si los resultados varían mucho, la confianza
es baja. Si son muy consistentes, la confianza es alta.

E) GUARDADO EN BASE DE DATOS
Todos los resultados se guardan en la base de datos PostgreSQL:
- La imagen original
- La imagen procesada (con fondo transparente)
- Las predicciones (alto, ancho, grosor, peso)
- Los niveles de confianza
- Los metadatos del lote

F) RESPUESTA AL USUARIO
El backend envía todos los resultados de vuelta al frontend, que los muestra
al usuario de forma clara y organizada. El usuario puede ver:
- Las medidas calculadas
- El nivel de confianza de cada medida
- Las imágenes procesadas
- Estadísticas del lote completo (promedios, totales, etc.)

Herramientas usadas:
- Segmentación: YOLOv8 (modelo preentrenado de Ultralytics)
- Predicción: PyTorch con modelos ResNet18 entrenados
- Procesamiento de imágenes: OpenCV, Pillow
- Normalización de datos: Scikit-learn
- Base de datos: PostgreSQL con Django ORM

Limitaciones actuales:
- El procesamiento es secuencial (una imagen a la vez), no paralelo, por lo que
  lotes grandes pueden tardar varios minutos
- Si YOLO no encuentra el grano después de varios intentos, la imagen se
  marca como fallida
- La precisión depende de la calidad de la imagen y el entrenamiento de los
  modelos
- El sistema actualmente funciona solo con CPU, no usa GPU, por lo que el
  procesamiento puede ser más lento que en sistemas con tarjetas gráficas


3. ENTRENAMIENTO DE MODELOS
-----------------------------

Los modelos de predicción no aparecen de la nada; deben ser entrenados con
datos reales. Este proceso toma horas pero se hace de forma asíncrona para no
bloquear el sistema.

A) PREPARACIÓN DEL DATASET
El sistema necesita un archivo CSV con las medidas reales de granos de cacao
y las imágenes correspondientes. Este dataset debe tener al menos:
- Nombre de archivo de imagen
- Alto real (en mm)
- Ancho real (en mm)
- Grosor real (en mm)
- Peso real (en gramos)

El sistema valida que todas las imágenes existan y que los datos sean válidos.

B) GENERACIÓN DE CROPS
Para cada imagen del dataset, el sistema usa YOLO para generar los crops
(granos aislados con fondo transparente). Si un crop no cumple con los
estándares de calidad (muy pequeño, muy deformado, etc.), el sistema lo
regenera automáticamente.

C) CREACIÓN DE ESCALADORES
Los valores reales (por ejemplo, peso puede estar entre 0.5g y 2.5g) se
normalizan para que el modelo pueda aprender mejor. Se crean "escaladores" que
convierten valores reales a valores normalizados y viceversa.

D) DIVISIÓN DEL DATASET
El dataset se divide en tres partes:
- 70% para entrenar el modelo
- 10% para validar durante el entrenamiento
- 20% para probar el modelo final

E) ENTRENAMIENTO
Se entrenan 4 modelos independientes (uno por cada medida) usando la
arquitectura ResNet18. Cada modelo se entrena durante hasta 150 épocas, donde
cada época significa que el modelo vio todas las imágenes de entrenamiento
una vez.

Durante el entrenamiento:
- Se aplican transformaciones aleatorias a las imágenes (rotaciones, cambios
  de brillo, etc.) para que el modelo aprenda a ser más robusto
- Se ajusta el learning rate (qué tan rápido aprende el modelo) de forma
  inteligente
- Se detiene automáticamente si el modelo deja de mejorar (early stopping)
- Se usa una función de pérdida robusta que es menos sensible a valores
  anómalos

F) EVALUACIÓN
Al final, se evalúan los modelos con el conjunto de prueba para medir qué tan
bien funcionan. Se calculan métricas como el error promedio, el error máximo,
y qué tan bien se ajustan los resultados.

G) GUARDADO
Los modelos entrenados se guardan en archivos para que el sistema de
predicción los pueda usar. También se guardan los escaladores correspondientes.

Todo este proceso se ejecuta en segundo plano usando Celery, por lo que el
usuario puede seguir usando el sistema normalmente mientras se entrena.

El usuario puede monitorear el progreso desde la interfaz, que actualiza cada
5 segundos mostrando el estado del entrenamiento (en progreso, completado,
fallido, etc.).

Herramientas usadas:
- PyTorch: Framework principal para deep learning
- ResNet18: Arquitectura de red neuronal preentrenada en ImageNet
- Scikit-learn: Para normalización de datos y evaluación
- Celery: Para ejecutar el entrenamiento en segundo plano
- Redis: Como mensajero entre Django y Celery
- Albumentations: Para aplicar transformaciones a las imágenes durante el
  entrenamiento

Limitaciones actuales:
- El entrenamiento se ejecuta solo en CPU, por lo que puede tardar 2-3 horas.
  Con GPU sería mucho más rápido (30-60 minutos)
- Windows tiene limitaciones con procesamiento paralelo, por lo que solo se
  procesa una imagen a la vez durante el entrenamiento
- El sistema requiere un dataset limpio y bien etiquetado. Si hay muchos
  errores en los datos, los modelos no aprenderán bien


4. GESTIÓN DE FINCAS Y LOTES
------------------------------

El sistema permite organizar los análisis por fincas y lotes. Una finca puede
tener múltiples lotes, y cada lote puede tener múltiples análisis de granos.

Esta información se guarda en la base de datos y permite:
- Agrupar análisis por lote
- Calcular estadísticas por lote o finca
- Generar reportes organizados
- Rastrear el origen de cada muestra

Herramientas usadas:
- Django ORM para gestionar la base de datos
- PostgreSQL para almacenar la información


5. GENERACIÓN DE REPORTES
--------------------------

El sistema puede generar reportes en PDF o Excel con toda la información de
los análisis. Los reportes incluyen:
- Resumen estadístico (promedios, totales, mínimos, máximos)
- Tablas con todos los análisis
- Gráficos de distribución
- Información de la finca y el lote

Herramientas usadas:
- ReportLab para generar PDFs
- OpenPyXL y XlsxWriter para generar archivos Excel


6. MONITOREO Y AUDITORÍA
-------------------------

El sistema registra todas las acciones importantes:
- Inicios de sesión de usuarios
- Análisis realizados
- Entrenamientos iniciados
- Cambios en configuración

Esto permite:
- Auditoría del sistema
- Identificación de problemas
- Seguimiento de uso

================================================================================
                    COMUNICACIÓN ENTRE COMPONENTES
================================================================================

El frontend y el backend se comunican mediante API REST, que es un estándar
ampliamente usado para este tipo de comunicación. Básicamente, el frontend
envía peticiones HTTP (GET para obtener datos, POST para enviar datos) y el
backend responde con datos en formato JSON.

Para autenticación, se usan tokens JWT que se generan cuando el usuario inicia
sesión y se envían en cada petición para identificar al usuario.

Para tareas largas (como el entrenamiento), se usa Celery con Redis. Django
"encola" la tarea en Redis, y un worker de Celery (que corre por separado) toma
la tarea y la ejecuta. El frontend puede consultar el estado de la tarea
mediante polling (consultas periódicas).

Opcionalmente, el sistema puede usar WebSockets para actualizaciones en tiempo
real, pero actualmente se usa principalmente polling por simplicidad y
confiabilidad.

================================================================================
                    FLUJO COMPLETO DE UN ANÁLISIS
================================================================================

1. Usuario inicia sesión → Frontend envía credenciales → Backend verifica →
   Genera token JWT → Frontend guarda token

2. Usuario sube imágenes → Frontend prepara FormData → Envía petición POST →
   Backend recibe imágenes

3. Para cada imagen:
   a) Backend usa YOLO para encontrar el grano
   b) Genera imagen con fondo transparente
   c) Pasa imagen a los 4 modelos de predicción
   d) Cada modelo predice su medida correspondiente
   e) Se calcula nivel de confianza
   f) Se guarda todo en base de datos

4. Backend agrupa resultados → Calcula estadísticas del lote → Responde al
   frontend

5. Frontend muestra resultados → Usuario puede ver medidas, confianzas,
   imágenes procesadas y estadísticas

Todo este proceso tarda aproximadamente 5-10 segundos por imagen, dependiendo
de la complejidad y el hardware disponible.

================================================================================
                    FLUJO COMPLETO DE UN ENTRENAMIENTO
================================================================================

1. Usuario inicia entrenamiento → Frontend envía configuración → Backend crea
   job → Encola tarea en Celery → Responde con job_id

2. Worker de Celery toma la tarea → Actualiza estado a "en progreso"

3. Pipeline de entrenamiento:
   a) Carga y valida dataset
   b) Genera/valida crops de todas las imágenes
   c) Normaliza los datos
   d) Divide en train/validation/test
   e) Entrena 4 modelos (uno por medida)
   f) Evalúa modelos con conjunto de prueba
   g) Guarda modelos y escaladores

4. Durante el entrenamiento, el frontend consulta el estado cada 5 segundos y
   muestra el progreso al usuario

5. Al finalizar, el sistema actualiza el estado a "completado" y guarda las
   métricas de evaluación

Este proceso puede tardar 2-3 horas en CPU, pero el usuario puede seguir usando
el sistema normalmente mientras ocurre.

================================================================================
                    CONSIDERACIONES Y LIMITACIONES
================================================================================

HARDWARE:
---------
- El sistema actualmente funciona mejor en CPU que en GPU porque está
  configurado para Windows, donde el soporte GPU es más complejo
- Para mejor rendimiento en producción, se recomienda usar Linux con GPU NVIDIA
- El procesamiento paralelo en Windows está limitado, lo que hace más lento
  el entrenamiento

PRECISIÓN:
----------
- La precisión de las predicciones depende directamente de la calidad del
  entrenamiento
- Cuantos más datos de entrenamiento, mejor será el modelo
- Las imágenes deben ser de buena calidad (buena iluminación, fondo
  contrastante, grano visible)

ESCALABILIDAD:
--------------
- El sistema puede manejar múltiples usuarios simultáneamente
- Para análisis batch grandes, el tiempo de procesamiento aumenta
  proporcionalmente
- El entrenamiento solo puede ejecutarse uno a la vez para evitar conflictos

MANTENIMIENTO:
--------------
- Los modelos deben reentrenarse periódicamente con nuevos datos para
  mantener la precisión
- La base de datos crece con el tiempo y puede necesitar limpieza periódica
- Los logs y crops de runtime deben limpiarse ocasionalmente para ahorrar
  espacio

================================================================================
                    RESUMEN
================================================================================

CacaoScan es un sistema completo que combina tecnologías web modernas con
inteligencia artificial para automatizar el análisis de granos de cacao.

El flujo general es:
1. Usuario se autentica
2. Usuario sube imágenes
3. Sistema procesa con IA (YOLO + modelos de predicción)
4. Resultados se guardan y muestran al usuario
5. Opcionalmente, se pueden entrenar nuevos modelos con más datos

Todo está diseñado para ser intuitivo para el usuario final, mientras que el
sistema maneja la complejidad técnica detrás de escena.

La arquitectura separa claramente las responsabilidades: el frontend maneja
la experiencia del usuario, el backend orquesta las operaciones, y los modelos
de IA hacen el trabajo pesado de análisis.

Para tareas largas como el entrenamiento, se usa procesamiento asíncrono para
no bloquear el sistema, permitiendo que múltiples usuarios lo usen
simultáneamente sin problemas.
